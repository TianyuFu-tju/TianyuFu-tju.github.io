EFDC源码开发环境配置

一、EFDC源码简介及下载途径

EFDC（Environmental Fluid Dynamics Code，环境流体力学代码）三维数学模型由美国弗吉尼亚海洋科学研究所（Virginia Institute of Marine Science, VIMS）的John M. Hamrick博士使用Fortran77语言开发。目前可用EFDC源码下载途径有以下三种：

  1. 国家环境保护环境影响评价数值模拟重点实验室
    地址：http://www.lem.org.cn/
    在该网站的“法规模型——地表水模型——EFDC模型”中可找到相关代码的下载地址。
  2. SNL-EFDC
    地址：https://sourceforge.net/projects/snl-efdc/
    SNL-EFDC是美国能源部的桑迪亚国家实验室（Sandia National Laboratory, SNL）在美国环保署（Environmental Protection Agency，EPA）发布的EFDC的基础上改进后的版本，修改的源码文件名加有前缀s_。
  3. EFDCPlus
    地址：https://github.com/dsi-llc/EFDCPlus
    EFDCPlus由美国DSI公司在原版EFDC上继续维护开发。

应当指出的是，美国环保署（Environmental Protection Agency）官网提供的EFDC模型（EFDC 1.01版本）已不包含源码，仅包含exe可执行文件及部分前后处理工具（部分含源码）、说明手册等内容。
由于EFDC模型采用Fortran77语言开发，因此其源码开发环境配置也将围绕能够支持Fortran77语法的集成开发环境（Integrated Development Environment, IDE）、编译器（Compiler）等进行尝试。下面将介绍四种常见的开发环境配置方案，分别为：

  1. CodeBlocks（IDE） & gfortran（Complier）
  2. SimplyFortran（IDE，集成gfortran complier）
  3. Visual Studio Community 2013（IDE） & Intel Visual Fortran 2013（Complier）
  4. Visual Studio Community 2022（IDE） & Intel oneAPI Base Toolkit（Complier）& Intel oneAPI HPC Toolkit（Complier）

二、开发环境搭建测试与比较

1. CodeBlocks & gfortran（失败）

该方案中采用CodeBlocks作为集成开发环境（经测试，CodeBlocks 17.12及后续版本已集成了gfortran编译器，无需额外安装），采用gfortran作为编译器进行Fortran77程序的开发。这一方案占用存储空间小、安装方便，因此也成为Fortran语言开发的主流环境配置方案。
经测试，该开发环境在编译EFDC源代码时会出现以下提示：

Error: FORM specifier in OPEN statement has invalid value ‘BINARY’

该错误会导致编译无法进行，不能生成有效的可执行文件。经过网上资料的查阅，出现该错误的原因是gfortran编译器对OPEN语句进行编译时，FORM参数仅支持“FORMATTED”或“UNFORMATTED”两种设置，而不支持“FORM = BINARY”的设置样式。实际上“FORM = BINARY”指的是以二进制形式读写文件，根据网上的资料推测可以将源码中“FORM = BINARY”全部替换为“FORM = UNFORMATTED”解决这一问题。

经对源码的查看得知，“FORM=BINARY”主要出现在输出为EFDC Explorer后处理准备的二进制文件部分中，不使用EFDC Explorer的话可将相关子程序全部删除后再尝试用gfortran编译。

2. SimplyFortran（失败）

该集成开发环境同样以其占用存储空间小、安装方便而受到Fortran开发者的推荐，但其失败的原因也与1相同——使用了gfortran作为编译器，因而推测其不适用于EFDC源码的开发。

3. Visual Studio Community 2013 & Intel Visual Fortran XE2013 SP1 update1（成功）

该方案也是使用人群较多的Fortran开发环境配置方案之一。该方案的优点是采用IVF作为编译器，不会产生上述两种方案中的错误，且应用广泛，功能全面，VS Communiy版本免费（需要微软Microsoft账户）；但也有着一些缺点，如版本已经过时，占用存储空间较多（10+GB），编译器Intel Visual Fortran 2013是收费软件等。

值得一提的是，为了避免安装中联网下载package出现问题，应尽量使用光盘镜像ISO文件进行VSCode2013的安装，注意package选择ISO中所包含的文件。尽量在VSCode2013安装完成后再进行IVF2013的安装，避免VSCode 2013载入IVF编译器出现问题。成功安装后打开VSCode 2013，依次点击菜单栏中的HELP——About Microsoft Visual Studio，若出现下图所示的IVF 2013加载项则证明安装成功。

3.2 其他问题

  1. 装IVF2013时可能需要手动指定许可证文件，在Activation步骤中先选择第3个选项“Choose Alternative Activation”，在下一页中再选择第2个选择“Use a License File”，后选择压缩包中的lic文件即可。
  2. 编译过程中可能遇到以下错误：

error #6634: The shape matching rules of actual arguments and dummy arguments have been violated.
error #8284: If the actual argument is scalar, the dummy argument shall be scalar unless the actual argument is of type character or is an element of an array that is not assumed shape, pointer, or polymorphic.

解决该错误的方法为
(1) 依次点击菜单栏中的DEBUG——Project Name Properties…，之后出现如图所示的窗口；
(2) 选择Fortran——Diagnastics选项，找到Check Routine Interfaces并设置为No。
目前最新版本DEBUG模式下似乎不能更改Check Routine Interfaces项的属性值了，直接换成Release模式运行。（2023.7.15）

  1. 编译过程中还可能出现代号为“general error c101008d”的错误，请查看文末参考资料中的（6）进行解决。
  2. 安装完成30天后，运行时可能出现“error #10052 could not checkout FLEXim license.”提示。推测该错误为IVF许可证过期导致的，请将IVF安装包中的intel_parallel_studio_xe_2013_update_3.lic文件放置在“C:\Program Files (x86)\Common Files\Intel\Licenses”或VS2013报错提示的路径下。（2021.8.24注）
  3. 为保证Release模式下编译出的exe文件能够在VSCode2013外独立运行，需要将DEBUG——Project Name Properties…——Fortran——Libraries下的Runtime Library改为“Multithreaded”。否则可能出现“应用程序无法正常启动(0xc000007b)”，或如下图所示的提示。（2022.1.21注）
    ![[20210708 EFDC源码开发环境配置.png]]
  4. 无论Debug模式还是Release模式，在首次运行建立好的模型时，都建议将DEBUG——Project Name Properties…——Fortran——Run-time下的Check Array and String Bounds设置为“Yes(/check:bounds)”，用来显示一些数组越界错误。（2022.1.21注）
  5. 采用自动并行编译设置编译出的exe文件在VSCode 2013中能够正常运行，而独立运行可能会提示“应用程序无法正常启动(0xc000007b)”，这是因为其依赖的dll文件与系统中的dll文件不符。可以将C:\Program Files (x86)\Common Files\Intel\Shared Libraries\redist\ia32\compiler文件夹下的libiomp5md.dll复制到C:\Windows\System32或exe文件所在文件夹中，即可正常运行。使用任务管理器查看CPU占用率可发现已经开启并行计算。（2022.1.31注）

三、优化过程记录

1. 编译过程优化初次尝试（记录于2022.1.30）

仅关注文件的执行速度情况下，对编译出的可执行文件体积大小没有要求，因此以速度最快为第一目标配置编译器。首先以Release模式为基础新建编译模式，再按照以下内容设置编译过程，编译出的可执行文件可获得较快的运行速度。
![[20210708 EFDC源码开发环境配置-1.png]]

  1. Optimization：优化级别设为/O3
  2. Loop Unroll Count：循环展开 保持空白来让编译器自行决定展开层数
  3. Parallelization：开启自动并行化。（可能会使计算变慢或出现错误） /Qparallel
  4. Prefetch Insertion：开启极端的数据预读 /Qopt-prefetch=3
  5. Interprocedural Optimizations：开启过程间优化（IPO） /Qipo
    ![[20210708 EFDC源码开发环境配置-2.png]]
  6. Enable Enhanced Instruction Set：开启强化指令集SSE3进行向量化 /arch:SSE3
    ![[20210708 EFDC源码开发环境配置-3.png]]
  7. Runtime Error Checking：关闭运行时的错误检查（包括数组越界检查） /check:none
    （20220404 警告！取消/check:bounds可能会引发一系列的问题，参见下文）
    ![[20210708 EFDC源码开发环境配置-4.png]]
  8. Runtime Library：将运行库改为Multithreaded（否则会提示“0xc000007b”错误）

2. 检查越界问题（/check:bounds）

首先贴出关于/check:bounds官方手册的解释：
![[20210708 EFDC源码开发环境配置-5.png]]
说明的主要内容如下：

开启“check bounds”选项将在编译时和执行时检查数组下标和字符串子链（?）的表达式。若表达式超出数组维度或字符串长度将会报错。
…在程序完成debug过程后，可以省略该设置来减小可执行文件的体积并稍微提高执行速度…

经实测/check:bounds的开启与否将会极大地影响EFDC的计算速度（3~4倍），以及可执行文件尺寸，开启时约20000 KB，关闭时约6000~7000 KB。根据官方手册的说明，推荐关闭。

3. 速度对比

通过R脚本调用可执行文件并计时，EFDC Plus v8.5使用8threads，EFDC-EPA使用单核。测试案例为于桥水库水动力模型，模拟时长为2006年1月1日-1月18日共计18天。以下为结果对比：

Version EFDC Plus v8.5 EFDC EPA
Threads 8 1
Elasped time (s) 99.72 425.81

按此速度估算，模拟1年在EFDC EPA下需要近2.4h，而EFDC Plus v8.5仅需0.56min（33.7min），提速近4倍。

然而EFDC Plus v8.5没有直接生成SURFCON.OUT等结果文件的subroutine，需要自行编入结果输出文件。计算结果有待对比。（2023.09.04）
